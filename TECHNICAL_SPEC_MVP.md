# Техническое задание: MVP автоматизации подбора аналогов недвижимости

## 1. Цель

Разработать MVP-скрипты для автоматизации подбора аналогов по объектам недвижимости и заполнения Excel-шаблонов оценки.

Система должна быть универсальной: не только для паркомест, но и для других типов недвижимости. Паркинги считаются первым реализуемым профилем, потому что под них уже есть рабочий `.xls`-шаблон.

MVP должен снижать ручную работу: собирать данные по объявлениям, сохранять доказательные скриншоты источников, нормализовать параметры объектов и заполнять только заданную область Excel-шаблона.

## 2. Поддерживаемые типы объектов

Архитектура должна поддерживать разные типы недвижимости через поле `property_type`.

Первые типы, которые нужно предусмотреть:

- `parking` - подземный паркинг / паркоместо;
- `apartment` - квартира;
- `commercial` - коммерческое помещение;
- `office` - офис;
- `retail` - торговое помещение;
- `warehouse` - складское помещение;
- `house` - дом / коттедж;
- `land` - земельный участок.

В MVP фактическая реализация может начинаться с `parking`, но код и структура данных не должны быть жестко зашиты только под паркинги.

## 3. Основной сценарий MVP

Пользователь передает:

- адрес объекта оценки;
- город;
- район или ориентир;
- название ЖК/БЦ/комплекса, если применимо;
- тип объекта недвижимости;
- параметры объекта оценки;
- путь к Excel-шаблону;
- профиль заполнения шаблона;
- 5 ссылок на объявления-аналогов либо параметры для поиска кандидатов.

Система:

1. Загружает параметры задачи.
2. Определяет тип объекта и профиль шаблона.
3. Собирает или читает данные по 5 аналогам.
4. Проверяет, что аналоги сопоставимы с объектом оценки.
5. Делает полноразмерные скриншоты страниц источников.
6. Нормализует данные в единый формат.
7. Заполняет только разрешенные ячейки Excel-шаблона.
8. Сохраняет заполненный файл в отдельную папку результата.
9. Формирует краткий отчет о внесенных данных, источниках и предупреждениях.

## 3.1. Зафиксированные решения клиента

- Финальный результат MVP: готовый отчет в формате Word (`.docx`, при необходимости `.doc`). PDF не обязателен: оценщик сможет конвертировать отчет в PDF самостоятельно.
- Рабочий режим по итогам месяца: система должна складывать готовые отчеты в папку периода, чтобы оценщик мог переслать их на печать в типографию.
- Старый отчет `.doc` можно один раз конвертировать в `.docx` и превратить в шаблон с плейсхолдерами.
- Входные документы могут приходить одним большим PDF-массивом или отдельными файлами по объектам. Если документы идут большим массивом, порядок внутри массива ожидается такой: сначала витяг, затем техпаспорт.
- Если в одном PDF найдено несколько объектов недвижимости, система должна сформировать отдельный отчет по каждому объекту. PDF читается/OCR-ится один раз, после чего для каждого найденного объекта создается отдельная папка с собственными `intake.json`, расчетной таблицей Excel, Word-отчетом и validation-файлами.
- Количество комнат для квартиры определяется по техпаспорту: количество строк с заполненной жилой площадью в столбце 6 экспликации помещений.
- Текущий Word-отчет является образцом/«козою» для нового отчета по аналогичному объекту, а не входными данными нового кейса.
- Полные реквизиты собственников берутся из существующего отчета/шаблона. Из витяга собственники используются в первую очередь для сверки.
- Для MVP нужно минимум 5 аналогов. Отдельный «огляд ринку» не входит в текущий расчетный сценарий и будет отдельным модулем.
- `Корегування на торг` на первом этапе не автоматизируется и не перезаписывается. В расчетной таблице строка торга должна сохраняться как в шаблоне или заполняться вручную оценщиком.
- Актуальный ориентир по торгу для Киева и Львова сейчас около 4,5%, но это не включается в автоматический расчет MVP.
- В финальный отчет вставляется читабельный скриншот объявления. Для архива можно хранить full-page screenshot, но в отчет нужно вставлять оптимизированное изображение с разумным весом.

## 4. Этапы реализации

### Этап 1: полуавтоматический режим

Пользователь вручную передает 5 ссылок на объявления.

Скрипт должен:

- открыть каждую ссылку;
- извлечь основные данные;
- сохранить полноразмерный скриншот всей страницы;
- подготовить структурированный `candidates.json`;
- проверить данные;
- заполнить Excel-шаблон.

Этот режим считается основным для первого MVP, потому что он быстрее в реализации и меньше зависит от изменений поисковых сайтов.

### Этап 2: поиск кандидатов

Скрипт ищет кандидатов по заданным параметрам на поддерживаемых источниках.

Для MVP поиск может быть ограничен несколькими источниками:

- DIM.RIA;
- Rieltor.ua;
- Real Estate Lviv (`real-estate.lviv.ua`);
- OLX, если страница доступна без блокировок;
- LUN, если доступна подходящая страница;
- сайты ЖК, БЦ, застройщиков или агентств, если они указаны пользователем.

## 5. Общие требования к аналогам

В подбор попадают только объекты, которые соответствуют всем условиям:

- тип сделки соответствует задаче: продажа или аренда;
- тип недвижимости соответствует объекту оценки;
- город и район сопоставимы с объектом оценки;
- есть цена или явно зафиксировано, что цена отсутствует;
- есть активная ссылка на источник;
- объявление не является дублем уже выбранного аналога;
- источник доступен для проверки через скриншот.

Для объектов в новостроях/комплексах дополнительно проверяется:

- наличие признаков ЖК, БЦ, комплекса или застройщика;
- стадия готовности или срок сдачи, если это важно для конкретного шаблона;
- класс объекта, если он используется в расчете.

## 6. Правила сопоставимости по типам

Для каждого `property_type` должен быть отдельный профиль фильтрации.

### 6.1. Паркинг

Подходит:

- подземный паркинг;
- паркоместо;
- машиноместо в ЖК/новострое;
- продажа, если задача на продажу.

Отбрасывается:

- аренда, если задача на продажу;
- гараж, если он не сопоставим с объектом оценки;
- наземная стоянка, если оценивается подземный паркинг;
- объект без признаков ЖК/новостроя, если в задаче требуется только новострой.

### 6.2. Квартира

Подходит:

- квартира того же или близкого класса;
- сопоставимое количество комнат или площадь;
- тот же город, район или сопоставимая локация;
- первичный или вторичный рынок согласно задаче.

Отбрасывается:

- комнаты;
- доли;
- апартаменты, если они юридически не сопоставимы;
- объекты с другой стадией готовности, если это влияет на расчет.

### 6.3. Коммерческая недвижимость

Подходит:

- помещения схожего назначения;
- сопоставимая площадь;
- сопоставимый тип входа, этаж, фасадность и трафик, если эти параметры указаны.

Отбрасывается:

- помещения другого назначения;
- объекты без цены;
- предложения с непонятной единицей цены.

### 6.4. Земельные участки

Подходит:

- участки с сопоставимым целевым назначением;
- сопоставимая площадь;
- сопоставимая локация и доступность коммуникаций.

Отбрасывается:

- участки с другим назначением;
- объявления без площади;
- объявления без понятной цены.

## 7. Единая структура данных аналога

Для каждого объекта сравнения нужно получить общий набор полей:

- `source_url` - ссылка на объявление;
- `source_name` - источник: DIM.RIA, Rieltor, OLX, LUN, сайт ЖК/БЦ и т.д.;
- `property_type` - тип недвижимости;
- `transaction_type` - продажа или аренда;
- `title` - заголовок объявления;
- `address` - адрес;
- `city` - город;
- `district` - район, если указан;
- `complex_name` - название ЖК/БЦ/комплекса, если указано;
- `area_m2` - площадь, м²;
- `price` - исходная цена;
- `currency` - валюта исходной цены;
- `price_usd` - цена в USD;
- `price_per_m2_usd` - цена за м² в USD, если применимо;
- `floor_or_level` - этаж/уровень, если указан;
- `rooms` - количество комнат, если применимо;
- `condition` - состояние/ремонт, если применимо;
- `building_class` - класс ЖК/БЦ/объекта, если известен;
- `purpose` - назначение объекта;
- `delivery_date` - срок сдачи/готовность, если применимо;
- `location_quality` - качественная оценка расположения для шаблона;
- `screenshot_path` - путь к полноразмерному скриншоту страницы;
- `collected_at` - дата и время сбора данных;
- `warnings` - список предупреждений по качеству данных.

Поля, не относящиеся к конкретному типу объекта, допускается оставлять пустыми.

## 8. Требования к скриншотам

Скриншоты источников обязательны для каждого аналога.

Требования:

- делать полноразмерный скриншот всей страницы для архива, а не только видимой области браузера;
- формат файла: `.png`;
- имя файла должно быть стабильным и понятным, например `01_dimria_parking_33795294.png`;
- скриншот должен сохраняться рядом с результатами задачи;
- на скриншоте должны быть видны цена, заголовок, адрес/комплекс и ключевые параметры, если они присутствуют на странице;
- если страница длинная, сохраняется именно full-page screenshot;
- если сайт блокирует часть контента, это фиксируется в `report.md`;
- если cookie/banner перекрывает данные, скрипт должен попытаться закрыть его перед скриншотом.
- для финального Word-отчета full-page screenshot может быть дополнительно обрезан/масштабирован/сжат до читабельного изображения с меньшим весом файла.

Технически full-page screenshots выполняются через Playwright:

```python
page.screenshot(path=screenshot_path, full_page=True)
```

## 9. Excel-шаблоны и профили заполнения

Система должна работать не с одним жестко заданным шаблоном, а с профилями заполнения.

Профиль шаблона описывает:

- путь к файлу шаблона;
- имя листа;
- разрешенный диапазон заполнения;
- какие поля объекта пишутся в какие ячейки;
- какие строки или ячейки запрещено перезаписывать;
- как вставлять hyperlinks;
- какие формулы должны сохраниться.
- какие строки расчета запрещено трогать на текущем этапе.

Для MVP строка `Корегування на торг` считается защищенной: система не должна менять ее автоматически, пока оценщик не утвердит финальную методику.

## 12. Работа с `.xls`

Так как текущий шаблон находится в старом формате `.xls`, для MVP используем Excel COM на Windows.

Причина:

- `openpyxl` корректно работает с `.xlsx`, но не с `.xls`;
- Excel COM позволяет сохранить форматирование, формулы, границы и hyperlinks исходного шаблона.

Основной инструмент:

- Python;
- `pywin32`;
- установленный Microsoft Excel.

Для будущих `.xlsx`-шаблонов можно добавить отдельный backend на `openpyxl`, но MVP должен надежно работать с текущим `.xls`.

## 15. Валидация перед заполнением

Перед записью в Excel скрипт должен проверить:

- найдено ровно 5 аналогов;
- у каждого аналога есть ссылка;
- у каждого аналога есть тип объекта;
- тип каждого аналога соответствует `target.property_type`;
- у каждого аналога есть цена или понятное предупреждение об отсутствии цены;
- у каждого аналога есть площадь, если она нужна для конкретного шаблона;
- каждый аналог проходит профиль фильтрации для своего типа недвижимости;
- скриншот создан и файл существует;
- целевой Excel-шаблон доступен;
- нужный лист существует;
- разрешенный диапазон заполнения существует;
- защищенные ячейки и формулы не будут перезаписаны.

Если критических данных не хватает, скрипт должен остановиться и вывести понятную ошибку.

## 16. Выходные файлы

После успешного запуска должны появиться:

- заполненный `.xls` или `.xlsx` в зависимости от исходного шаблона;
- финальный Word-отчет `.docx` или `.doc`;
- `candidates.json` с нормализованными данными;
- `report.md` с кратким описанием источников и предупреждений;
- папка `screenshots/` с 5 полноразмерными скриншотами;
- папка `report_images/` с оптимизированными изображениями для вставки в Word-отчет, если они отличаются от архивных full-page screenshots.

## 17. Нефункциональные требования

- Скрипт должен работать локально на Windows.
- Скрипт не должен зависеть от подписки OpenAI.
- OpenAI/LLM можно использовать позже только как опциональный модуль для улучшения извлечения данных.
- Все исходные ссылки и скриншоты должны сохраняться для аудита.
- Исходный шаблон не перезаписывать; всегда создавать копию результата.
- Ошибки должны быть понятными для пользователя без технического погружения.
- Код должен быть расширяемым через новые `property_type` и `template_profile`, а не через копирование отдельных скриптов под каждый тип объекта.

## 18. Критерии готовности MVP

MVP считается готовым, если:

1. Пользователь передает 5 ссылок на объявления.
2. Скрипт определяет и валидирует тип объекта.
3. Скрипт делает 5 полноразмерных скриншотов страниц.
4. Скрипт формирует `candidates.json`.
5. Скрипт заполняет только разрешенный диапазон шаблона.
6. Формулы и защищенные ячейки сохраняются.
7. В ячейках источников есть активные ссылки.
8. Исходный файл шаблона остается без изменений.
9. Итоговый файл открывается в Microsoft Excel без ошибок.
10. Визуально заполненный блок соответствует исходному шаблону.
11. Добавление нового типа объекта требует нового профиля и правил валидации, а не переписывания всей системы.
12. Система генерирует Word-отчет по `.docx`-шаблону с подставленными переменными.
13. Отчетные изображения объявлений читабельны и не раздувают файл без необходимости.
14. Если PDF содержит несколько объектов, система формирует пакет отчетов: один объект = одна папка результата = один Word-отчет и один Excel-расчет.

## 19. Что не входит в MVP

На первом этапе не реализуем:

- полностью автономный поиск по всем сайтам;
- обход капч и антибот-защиты;
- автоматическое определение класса ЖК/БЦ с высокой юридической точностью;
- автоматическое геокодирование и расчет расстояний;
- генерацию полноценного оценочного отчета в Word/Google Docs;
- ручное редактирование скриншотов;
- массовую промышленную обработку десятков/сотен независимых кейсов за один запуск; batch-режим внутри одного клиентского PDF входит в MVP;
- поддержку всех типов недвижимости на уровне production-качества.

Эти функции можно добавить после того, как базовый сценарий со ссылками, скриншотами, нормализацией данных и Excel-шаблоном будет стабильно работать.
